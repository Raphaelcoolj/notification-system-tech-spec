Notification System
at Scale

Architecture for a reliable, multi-channel notification service supporting push, SMS, and email delivery for 1M+ users — with no duplicates, no missed sends, and graceful provider failover.

Channels Push · SMS · Email Scale 1M+ users

Overview

This document specifies the architecture of a notification delivery system capable of reliably sending push notifications, SMS, and emails to over 1 million users. The system prioritizes exactly-once delivery, graceful degradation when third-party providers fail, and horizontal scalability.

Core Goals
  • No duplicate notifications sent
  • No missed sends under normal load
  • Automatic failover when providers fail
  • Full audit trail per notification
Non-Goals
  • Real-time delivery under <100ms
  • Building our own SMS/push infra
  • Rich in-app notification UI
  • Two-way messaging (replies)

Architecture Overview

The system uses an event-driven pipeline. Upstream services publish notification events to a message queue. Channel workers consume those events and dispatch to third-party providers.

[ Any Service ] ──publish──▶ [ Message Queue ] ──consume──▶ [ Channel Workers ] (auth, trades, (Kafka / SQS) Push Worker alerts, etc.) SMS Worker Email Worker │ │ ▼ ▼ [ Notification DB ] [ Provider APIs ] (status tracking) FCM / APNs (push) Twilio / Termii (SMS) SendGrid / SES (email) │ ▼ [ Dead Letter Queue ] (failed deliveries → retry / alert)

Key Components

Message Queue

Kafka (preferred for scale) or AWS SQS. Decouples producers from consumers. Partitioned by user_id for ordering guarantees where needed.

Channel Workers

Stateless Node.js services. Horizontally scalable. Each worker owns one channel (push/SMS/email) and is deployed independently.

Notification DB

PostgreSQL table tracking every notification's lifecycle state. Source of truth for idempotency checks and reconciliation.

Dead Letter Queue

Catches messages that fail after all retries. Triggers alerting to on-call. Supports manual replay once the issue is resolved.

Data Model

notifications table

CREATE TABLE notifications (
  id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  idempotency_key VARCHAR(255) UNIQUE NOT NULL,  -- prevents duplicates
  user_id         UUID         NOT NULL,
  channel         VARCHAR(10)  NOT NULL,       -- 'push' | 'sms' | 'email'
  template_id     VARCHAR(100) NOT NULL,
  payload         JSONB        NOT NULL,
  status          VARCHAR(20)  NOT NULL DEFAULT 'QUEUED',
  provider        VARCHAR(50),                   -- which provider was used
  provider_ref    VARCHAR(255),                  -- provider's message ID
  attempt_count   INT          DEFAULT 0,
  last_error      TEXT,
  created_at      TIMESTAMPTZ  DEFAULT now(),
  sent_at         TIMESTAMPTZ,
  delivered_at    TIMESTAMPTZ
);

-- Indexes for common query patterns
CREATE INDEX idx_notifications_user_status
  ON notifications (user_id, status, created_at DESC);

CREATE INDEX idx_notifications_status_created
  ON notifications (status, created_at)
  WHERE status IN ('QUEUED', 'FAILED');

Status Lifecycle

01
QUEUED
Event received and persisted. Message published to queue.
02
PROCESSING
Worker has consumed the message and is making the provider API call.
03
SENT
Provider accepted the request and returned a success response.
04
DELIVERED
Provider webhook confirmed delivery to the device/inbox.
05
FAILED
All retry attempts exhausted. Moved to DLQ and triggers an alert.

Idempotency & Duplicate Prevention

Every notification is assigned an idempotency_key by the publishing service, constructed as:

// Format: {event_type}:{user_id}:{entity_id}:{date}
const key = `withdrawal_complete:${userId}:${txnId}:${date}`;

Before any worker dispatches to a provider, it performs an upsert:

INSERT INTO notifications (idempotency_key, user_id, channel, ...)
VALUES ($1, $2, $3, ...)
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING id, status;

-- If status is already SENT or DELIVERED → skip. Do not call provider again.
Important: The idempotency check happens inside a database transaction before any external API call. If the row already exists with status SENT or DELIVERED, the worker acknowledges the queue message and exits — no provider call is made.

Provider Failover & Degradation

Each channel has a primary and fallback provider. A circuit breaker per provider controls automatic routing.

Channel Primary Fallback Circuit Breaker Threshold
Push FCM (Android) / APNs (iOS) OneSignal 5 failures / 30s window
SMS Twilio Termii / Africa's Talking 5 failures / 30s window
Email SendGrid AWS SES 10 failures / 60s window

Circuit Breaker States

CLOSED (normal) │ 5 consecutive failures ▼ OPEN (failing — route to fallback) │ 60s cooldown ▼ HALF-OPEN (test one request) │ success failure ▼ ▼ CLOSED OPEN (reset timer)

Circuit breaker state is stored in Redis for shared access across all worker instances. TTL on the OPEN state key acts as the automatic reset timer.

Retry Strategy

Transient failures (timeouts, 5xx responses) are retried with exponential backoff. Permanent failures (invalid phone number, unsubscribed email) are not retried.

// Retry schedule (attempt → delay)
const RETRY_DELAYS = [
  30,    // 30 seconds
  120,   // 2 minutes
  600,   // 10 minutes
];

async function handleFailure(notification, error) {
  const isPermanent = isPermanentError(error); // 400, invalid recipient, etc.

  if (isPermanent || notification.attempt_count >= 3) {
    // Move to DLQ, mark FAILED, trigger alert
    await sendToDLQ(notification);
    await markFailed(notification.id, error.message);
  } else {
    const delay = RETRY_DELAYS[notification.attempt_count];
    await requeueWithDelay(notification, delay);
  }
}

Scalability Considerations

Horizontal Scaling

Workers are stateless and scale independently per channel. During peak load (e.g., market open), the SMS worker can scale to 20 instances while push stays at 5.

Queue Partitioning

Kafka topics are partitioned by user_id % N. Ensures per-user ordering and even load distribution across consumer instances.

Rate Limiting

Per-user rate limits enforced in Redis (notify:{userId}:{channel}:{window}) to prevent notification spam and respect provider limits.

Batch Processing

Broadcast notifications (e.g., platform announcements) use a fan-out service that batches DB writes and publishes to queue in chunks of 1,000.
